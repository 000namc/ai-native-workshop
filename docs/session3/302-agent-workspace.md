# 302. 작업환경 설계 — 에이전트가 일하기 편한 환경

> Session 3: 코딩 에이전트 기본

## 이것은 무엇인가요?

에이전트가 효율적으로 일할 수 있도록 프로젝트 환경을 설계하는 방법입니다.

## 왜 중요한가요?

- 잘 정리된 환경 = 에이전트의 실수가 줄어듭니다
- 명확한 구조 = 원하는 결과를 더 잘 얻을 수 있습니다

에이전트는 사람과 비슷합니다. 깔끔하게 정리된 책상에서 일을 시키면 잘하고, 어지러운 책상에서 일을 시키면 헷갈려합니다.

## 핵심 원칙

### 1. 폴더 구조를 명확하게

```
my-project/
├── CLAUDE.md          ← 에이전트가 가장 먼저 읽는 파일
├── src/               ← 소스 코드
├── build/             ← Docker 설정
├── docs/              ← 문서
└── package.json       ← 프로젝트 설정
```

### 2. 작업 공간을 잘 분리하기

에이전트에게 "데이터 조회 코드 수정해줘"라고 시켰을 때, 해당 코드만 바라보면 되는 환경이 이상적입니다. 프론트엔드 코드, 백엔드 코드, 설정 파일이 한 폴더에 뒤섞여 있으면 에이전트는 관계없는 코드까지 읽고 파악하느라 혼란스러워합니다.

```
❌ 성격이 다른 코드가 섞여 있는 구조

my-project/
├── timer.html
├── timer.js
├── server.py
├── database.py
├── Dockerfile
└── styles.css
```

```
✅ 역할별로 분리된 구조

my-project/
├── src/
│   ├── routes.py        ← API (백엔드)
│   ├── models.py        ← DB (데이터)
│   ├── static/          ← CSS, JS (프론트엔드)
│   └── templates/       ← HTML (프론트엔드)
└── build/
    └── Dockerfile       ← 환경 설정
```

분리가 잘 되어 있으면 "models.py 수정해줘"라고만 해도 에이전트가 다른 코드에 영향 없이 정확하게 작업할 수 있습니다.

### 3. 일단 시켜보고, 점점 구체적으로

처음부터 완벽하게 설계할 필요는 없습니다. 오히려 **일단 시켜보는 것**이 좋은 출발점입니다.

```
1단계: "타이머 앱 만들어줘"
       → 에이전트가 만든 결과물을 보면서 감을 잡는다

2단계: 결과를 보고 구체적으로 다시 요청한다
       → "타이머는 25분/5분 고정, 세션 기록은 DB에 저장, 오늘 기록만 목록으로 보여줘"

3단계: 작업을 쪼개서 하나씩 진행한다
       → "먼저 타이머 UI만 만들어줘"
       → "이제 DB 연동을 추가해줘"
       → "Docker 설정을 만들어줘"
```

핵심은 결과물을 보면서 **작업을 쪼개고, 요구사항을 명확하게 다듬어가는 과정**입니다. 이 과정을 반복하면서 자연스럽게 좋은 프로젝트 구조가 만들어집니다.

## 정리

- 폴더 구조를 명확하게, 성격이 다른 코드는 분리하기
- 에이전트가 필요한 코드만 바라볼 수 있는 환경이 최고
- 일단 시켜보고, 결과를 보면서 작업을 쪼개고 구체화하기
